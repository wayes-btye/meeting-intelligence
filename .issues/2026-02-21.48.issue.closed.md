## Problem
The chunking strategy is invisible to users in two places:
1. **Upload form** — radio buttons for Naive / Speaker Turn show no description or parameters
2. **Meetings list** — no indication of which strategy was used when a meeting was ingested

The retrieval strategy section (chat UI) already shows a parameter info block. Chunking should match this pattern.

## Solution

### Part 1 — Upload form parameter info block (frontend only)
Below the Chunking Strategy radio buttons, show a dynamic info block that updates on selection:

- **Naive selected:** `chunk_size: 500 tokens · overlap: 50 tokens`
  Splits the transcript into fixed-size windows regardless of speaker boundaries.
- **Speaker Turn selected:** `max_chunk_size: 500 tokens · split: on speaker change`
  One chunk per continuous speaker segment — preserves conversational context.

Values come from `src/config.py` (chunk_size: 500, chunk_overlap: 50). Pure frontend change, no API calls needed.

### Part 2 — Store chunking strategy on ingest (migration + backend)
Add `chunking_strategy` column to the `meetings` table, populated on ingest:

```sql
ALTER TABLE meetings ADD COLUMN chunking_strategy text;
```

Update `store_meeting()` in `src/ingestion/storage.py` to write the strategy.

### Part 3 — Show chunking strategy in meetings list (frontend)
Display the stored strategy on each meeting card/row in the meetings list, so users can see at a glance which strategy was used when a meeting was processed.

## Out of scope for this issue
- Tweakable chunk_size / chunk_overlap sliders (Phase 2 — separate issue if needed)

## Effort
Medium — one Supabase migration + small backend change + two frontend touches (upload form info block + meetings list column).

**Migration note:** Apply after confirming no other schema changes are pending. Current migration sequence: initial schema → `20260302000000_add_user_id_to_meetings.sql` (#71) → this.
