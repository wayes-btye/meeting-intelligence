## Problem
All retrieval parameters are hardcoded and invisible to the user. For a demo/assessment this is a missed opportunity to show how the system works.

## Current hardcoded values
| Parameter | Value | Location |
|-----------|-------|----------|
| Embedding model | \`text-embedding-3-small\` | \`search.py\` |
| \`match_count\` | 10 | \`search.py:25\` |
| \`vector_weight\` | 0.7 | \`search.py:55\` |
| \`text_weight\` | 0.3 | \`search.py:56\` |

## Phase 1 — Display only (low effort)
Show a parameter info block below the retrieval strategy radio buttons:
- Semantic: \`model: text-embedding-3-small · top_k: 10 · metric: cosine\`
- Hybrid: \`vector_weight: 0.7 · text_weight: 0.3 · top_k: 10\`

Add a tooltip (\`?\` icon) explaining each strategy in plain English.

## Phase 2 — Tweakable sliders (medium effort)
Expose \`vector_weight\`, \`text_weight\`, and \`match_count\` as UI controls. Requires the API to accept these as optional query params (currently hardcoded in \`search.py\`).

## Also in scope
- Show meeting title on source cards when querying "All meetings" (see #43)
- Show \`model\` name and token \`usage\` from API response below the answer
- Empty-state message when structured lookup returns no chunks: "Answer from structured database — no transcript chunks retrieved"

## Effort
Phase 1: Low — pure frontend display.
Phase 2: Medium — API param threading + slider components.
