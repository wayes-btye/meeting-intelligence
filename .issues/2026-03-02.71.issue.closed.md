## Problem

All logged-in users currently share the same data — every meeting is visible to every account. This must be isolated so each user only sees the meetings they uploaded.

Auth infrastructure already exists (issue #52): Supabase email/password login, Next.js middleware protecting routes. What's missing is the data layer enforcement.

## Solution

Thread the authenticated user's identity from the Supabase session JWT through to the FastAPI backend, and store `user_id` on every meeting row. All queries filter by the authenticated user. No changes needed to the login UI.

Users are created manually in the Supabase dashboard (Authentication → Users → Add user). No self-registration. The system handles the rest automatically once a user logs in and starts uploading.

## Implementation

**DB migration (write, do NOT apply without user instruction):**

```sql
ALTER TABLE meetings ADD COLUMN IF NOT EXISTS user_id uuid REFERENCES auth.users(id);
CREATE INDEX IF NOT EXISTS idx_meetings_user_id ON meetings(user_id);

-- Assign existing 10 meetings to reviewer@example.com
UPDATE meetings
SET user_id = (SELECT id FROM auth.users WHERE email = 'reviewer@example.com')
WHERE user_id IS NULL;
```

**Backend — JWT auth dependency (`src/api/auth.py`, new file):**

FastAPI dependency that reads `Authorization: Bearer <token>` header, validates the Supabase JWT using `SUPABASE_JWT_SECRET` (add to `.env` and `src/config.py`), and returns the user's UUID (`sub` claim). Raises HTTP 401 if missing or invalid.

```python
async def get_current_user_id(authorization: str = Header(...)) -> str:
    # validate JWT, return sub (UUID string)
```

**Backend — thread user_id through all write/read paths:**

1. `POST /api/ingest` — inject `user_id` from dependency, pass to `ingest_transcript()` → `store_meeting()`, store in `meetings.user_id`
2. `GET /api/meetings` — filter by `user_id` from dependency
3. `GET /api/meetings/{id}` — verify row belongs to authenticated user (404 if not)
4. `DELETE /api/meetings/{id}` — verify ownership before delete
5. `POST /api/query` — get meeting IDs belonging to user, scope search to those

**Frontend — pass JWT to API on every request (`frontend/lib/api.ts`):**

Read the Supabase session access token from the browser client and include it in all API requests as `Authorization: Bearer <token>`. The frontend already manages the Supabase session from issue #52.

```typescript
const { data: { session } } = await supabase.auth.getSession()
// Include in every apiFetch call:
headers: { 'Authorization': `Bearer ${session?.access_token}` }
```

**Config:**

Add `SUPABASE_JWT_SECRET` to `src/config.py` Settings and `.env.example`. Value is in Supabase dashboard → Settings → API → JWT Secret.

## Additional fix: JWT verification via JWKS (commit `4e0dc2e`)

Supabase's dashboard showed a warning: the legacy HS256 JWT secret has been migrated to the new JWT Signing Keys system (ES256 / ECDSA P-256). The implementation in the first two commits used `algorithms=["HS256"]` with the static secret — this would have returned 401 on every real login token in production.

Fixed by switching to `PyJWKClient` which fetches the public signing keys from:
```
https://qjmswgbkctaazcyhinew.supabase.co/auth/v1/.well-known/jwks.json
```

Key benefits:
- Works correctly with Supabase's current ES256 tokens
- Handles future key rotation automatically — no manual secret updates
- `SUPABASE_JWT_SECRET` removed from config entirely — one fewer env var to manage

1. `supabase db push --linked` — apply migration from main workspace
2. Merge PR #72 — no additional env vars needed beyond what's already set
