## Summary

- Adds per-user data isolation so every user only sees and can modify their own meetings
- FastAPI validates the Supabase JWT from `Authorization: Bearer <token>` header on all endpoints; unauthenticated requests receive 422 (missing header) or 401 (invalid token)
- `user_id` is stored on every new meeting row; all reads/writes filter by the authenticated user's UUID
- Frontend sends the Supabase session access token on every API request

### P1 — Empty JWT secret allows forged tokens

**File:** `src/api/auth.py:23-28` | **Codex flagged this too**

`supabase_jwt_secret` defaults to `""` in `src/config.py:23`. PyJWT with `HS256` will happily sign and verify a token using an empty string as the secret. If `SUPABASE_JWT_SECRET` is not set in the environment, an attacker can forge tokens for any `sub` (user UUID) by signing with the empty string — and the `jwt.decode()` call at line 24 will accept them.

The fix is a fail-closed guard before the decode:

```python
if not settings.supabase_jwt_secret:
    raise HTTPException(status_code=503, detail="Auth service not configured")
```

### P1 — Structured query path leaks data across users

**File:** `src/api/routes/query.py:39-50` | **Codex flagged this too**

The `STRUCTURED` branch calls `lookup_extracted_items()` from `src/retrieval/router.py:130-154` with no user context whatsoever. That function queries `extracted_items` with only optional `meeting_id` and `item_type` filters — no `user_id`. The result:

- "What were the action items?" (no `meeting_id`) → returns action items from **every user's meetings** globally
- "What were the action items?" with a foreign `meeting_id` → returns extracted items from a meeting the caller doesn't own

This completely defeats the isolation goal for the structured path.

`lookup_extracted_items` needs a `user_id` parameter that joins through `meetings.user_id`, and the call site in `query.py:40` needs to pass it. The simplest SQL-level fix is:

```python
# router.py
def lookup_extracted_items(
    meeting_id: str | None = None,
    item_type: str | None = None,
    user_id: str | None = None,  # add this
) -> list[dict[str, Any]]:
    client = get_supabase_client()
    # Filter extracted_items to this user's meetings via a join or subquery
    if user_id:
        user_meeting_ids = [
            r["id"] for r in client.table("meetings")
                .select("id").eq("user_id", user_id).execute().data
        ]
        if not user_meeting_ids:
            return []
        query = client.table("extracted_items").select("*").in_("meeting_id", user_meeting_ids)
    else:
        query = client.table("extracted_items").select("*")
    ...
```

And in `query.py`:
```python
items = lookup_extracted_items(
    meeting_id=request.meeting_id,
    item_type=routed.item_type,
    user_id=user_id,  # add this
)
```

### P2 — Python-side search filter is a rough heuristic

**File:** `src/retrieval/search.py:68, 117`

The `fetch_count = match_count * 3` multiplier for fetching extra results before Python-side user filtering is an approximation. If the database has many meetings from other users and the user has few, or if the top-ranked chunks all belong to other users, the returned set could be empty or much shorter than `match_count` even when sufficient user-relevant content exists.

This is an architectural trade-off (pushing user filtering to the DB RPC would be cleaner). It's acceptable for now, but worth tracking. Consider raising the multiplier or adding a note about this limitation if it shows up in practice.

---

### P2 — `get_meeting` does an unnecessary two-step DB check

**File:** `src/api/routes/meetings.py:63-74`

The `GET /api/meetings/{meeting_id}` handler fetches by ID without a user filter, then post-checks ownership:

```python
result = client.table("meetings").select("*").eq("id", meeting_id).execute()
...
if m.get("user_id") != user_id:
    raise HTTPException(status_code=404, ...)
```

This reads the full row (including `raw_transcript`) before confirming ownership. Filtering at the query level — `.eq("id", meeting_id).eq("user_id", user_id)` — would return nothing for a non-owned meeting without fetching the row content, and the existing `if not detail_rows` check would fire naturally. Not a security bug (the 404 result is correct) but cleaner and cheaper.

---

### P3 — Migration: `user_id` column is nullable

**File:** `supabase/migrations/20260302000000_add_user_id_to_meetings.sql:2`

The column is added without a `NOT NULL` constraint. Since the UPDATE on line 6 runs in the same migration, any existing rows with no matching `reviewer@example.com` user would silently remain `NULL` after migration. If a new meeting is ever stored without a `user_id` due to a backend bug, it would be invisible to all users (filtered out by `user_id = ?`) with no error. A `NOT NULL` constraint after the backfill would prevent this class of silent data loss.

---

### P3 — `deleteMeeting` duplicates auth header logic

**File:** `frontend/lib/api.ts:134-148`

`deleteMeeting` does a raw `fetch` instead of going through `apiFetch`. This is technically because `apiFetch` calls `.json()` which would fail on the 204 No Content response. The fix would be to make `apiFetch` handle empty bodies gracefully (e.g. return `undefined` for 204). As written, this creates two places where auth header logic lives — if token refresh logic is ever added to `apiFetch`, `deleteMeeting` would need a manual update too.

---

### P3 — JWT error detail exposes internal token diagnostics

**File:** `src/api/auth.py:32`

```python
raise HTTPException(status_code=401, detail=f"Invalid token: {exc}") from exc
```

PyJWT exception messages include specifics like `"Signature has expired"` vs `"Signature verification failed"`. While not a critical leak, returning a generic `"Invalid token"` message to callers is better practice (attackers learn less about whether they need to fix expiry vs re-sign).

## Fix: JWT verification switched to JWKS endpoint (commit `4e0dc2e`)

**Root cause:** Supabase has migrated this project from the legacy HS256 symmetric secret to ES256 (ECDSA P-256) JWT Signing Keys. The `algorithms=["HS256"]` + static secret approach would have silently failed to verify any real user login token — every request would have returned 401 in production.

**What changed:**
- `auth.py` now uses `PyJWT.PyJWKClient` pointed at `SUPABASE_URL + /auth/v1/.well-known/jwks.json`
- Signing key is fetched once and cached; key rotation is handled automatically
- `algorithms=["ES256", "RS256"]` covers current Supabase ES256 and any future algorithm change
- `SUPABASE_JWT_SECRET` removed from `src/config.py` and `.env.example` — no static secret to manage

**No env var setup required in Cloud Run or Vercel** — one less secret to configure. The JWKS URL is derived from `SUPABASE_URL` which is already set.

This is fail-closed (no data leak) but the 401 is unhelpful when the real problem is a misconfigured URL.
