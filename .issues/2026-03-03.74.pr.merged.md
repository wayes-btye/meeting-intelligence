## Summary

Implements contextual retrieval per [Anthropic research](https://www.anthropic.com/news/contextual-retrieval) — before embedding each chunk, a Claude Haiku call generates a 1-2 sentence document-level context summary that is prepended to the chunk text. The original `chunk.content` is unchanged; only the text sent to the OpenAI embedding API is enriched.

- **`generate_chunk_context(chunk, meeting_title)`** — calls `claude-haiku-4-5-20251001` to generate per-chunk retrieval context (~$0.001/chunk)
- **`embed_chunks_with_context(chunks, meeting_title)`** — drop-in replacement for `embed_chunks()` that enriches text before embedding
- **`PipelineConfig.contextual_retrieval: bool = False`** — off by default, opt-in per ingest
- **`/api/ingest`** — accepts new `contextual_retrieval` (bool Form field); wires through to `ingest_transcript()`
- **17 mocked unit tests** in `tests/test_contextual_retrieval.py` covering both functions and pipeline routing

Overall the implementation is clean and well-structured. The design is sound: the opt-in flag, the drop-in function signature, and the test coverage are all solid. Two issues need attention before this could go to production (confirmed by Codex too).

### P1 — Unbatched embedding calls (performance regression)

**`src/ingestion/embeddings.py:103`**

`embed_chunks_with_context` calls `embed_texts([enriched_text])` once per chunk inside a loop:

```python
for chunk in chunks:
    context = generate_chunk_context(chunk, meeting_title)
    enriched_text = f"{context}\n\n{chunk.content}"
    embedding = embed_texts([enriched_text])[0]   # ← one API call per chunk
```

The existing `embed_chunks()` makes **one batched call** for all chunks. With contextual retrieval enabled, a meeting with 20 chunks now makes 20 separate OpenAI embedding API calls instead of 1. This is a real latency and rate-limit issue — OpenAI's batch endpoint is orders of magnitude more efficient.

Fix: collect all enriched texts first, then embed in one call:

```python
enriched_texts = [
    f"{generate_chunk_context(c, meeting_title)}\n\n{c.content}"
    for c in chunks
]
embeddings = embed_texts(enriched_texts)
return list(zip(chunks, embeddings, strict=True))
```

Note: the test at `tests/test_contextual_retrieval.py:189` (`test_handles_multiple_chunks`) asserts `mock_embed.call_count == 2`, which validates the current broken behaviour. That test would need updating alongside this fix. ### Minor observations

**`PipelineConfig` field is orphaned** — `pipeline_config.py:36` adds `contextual_retrieval: bool = False` to `PipelineConfig`, but `ingest_transcript()` takes a plain `bool` parameter and never reads from a `PipelineConfig` object. The field is correct as documentation/intent, but nothing actually reads it to drive behaviour. This is a pre-existing pattern (same as `ChunkingStrategy`), so not blocking, but worth noting.

**Client instantiation per call** — `generate_chunk_context()` creates a new `anthropic.Anthropic()` client on every call (line 59), and `embed_texts()` creates a new `OpenAI()` client every call (line 23). With the loop as-is, that's N client objects per meeting. Once the batching fix (P1) is applied, this drops back to one client per pipeline run. Not blocking, but module-level or injected clients would be cleaner long-term.

**`max_tokens=100` is tight** — asking for "1-2 sentences" with a 100-token cap is borderline for verbose meeting titles or longer Claude responses. 150–200 tokens would give headroom without meaningful cost impact at Haiku pricing.

**`call_args` extraction in tests is fragile** — `test_passes_meeting_title_in_prompt` at line 53:
```python
messages = call_kwargs[1]["messages"] if call_kwargs[1] else call_args[0][2]
```
`call_kwargs[1]` is always truthy if keyword args were passed (it's a dict, not a value), so the `else` branch never fires. Using `call_args.kwargs["messages"]` would be more robust.
